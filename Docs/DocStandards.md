# 文档规范

## 目的

本规范用于指导所有开发 agent 编写和维护项目文档，确保文档精简、准确、易于查找。

## 精简原则

- **只记录必要信息**：脚本职责、公开 API（属性/方法/事件）、系统间依赖关系
- **禁止冗长内容**：不写实现细节、不贴大段代码、不保留过时历史说明
- **一句话描述优先**：每个脚本用一句话说明职责，API 用表格列出签名和说明

## 格式统一

每个系统文档固定包含以下四个章节：

### 1. 概述
- 系统职责一句话描述
- 所属脚本文件夹路径

### 2. 脚本清单
- 表格格式：脚本名 | 类型（MonoBehaviour/ScriptableObject/静态类/接口/枚举） | 一句话职责

### 3. API 速查
- 按脚本分组
- 仅列出公开属性、公开方法、公开事件
- 格式：`签名` — 说明

### 4. 依赖关系
- 本系统依赖哪些其他系统
- 哪些系统依赖本系统

## 维护要求

每次代码修改涉及文档更新时，必须同时完成：

1. **更新对应系统文档**：修改涉及的脚本说明、API 变更
2. **更新 INDEX.md**：
   - 如有脚本新增/删除/迁移，更新脚本目录树
   - 如有系统变更，更新系统一览表和快速检索表
3. **如新增系统**：创建新的系统文档，遵循上述四章节格式

## 命名规范

- 文档文件名：PascalCase 英文，如 `CoreSystem.md`、`FishCardSystem.md`
- 脚本路径引用：统一使用相对于 `Assets/` 的路径，如 `Script/Core/GameManager.cs`

## 废弃处理

- 废弃脚本迁移至 `Script/Abandoned/`，代码中添加 `[System.Obsolete]` 标注
- 废弃脚本不需要文档描述，从对应系统文档中移除
- 不保留过时文档，直接删除
